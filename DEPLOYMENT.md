# Deployment Guide — Vision Digital Lab

Deploy the stack to free hosting with your domain **visiondigitallab.com**.

```
visiondigitallab.com (+ www)   ─►  Vercel    (Next.js frontend)
api.visiondigitallab.com        ─►  Render    (Spring Boot backend, Docker)
                                ─►  Supabase  (Postgres — already live)
```

> No GitHub Actions needed: **Vercel and Render auto-build and redeploy on every
> `git push` to `main`.**

> ⚠️ **Never put real secrets in this file** (the repo is public). All credentials
> below are placeholders — use the real values from your git-ignored
> `backend/config/application.yml` and Supabase dashboard, entered directly into
> the Render/Vercel dashboards.

---

## 0. Prerequisites
- Repo pushed to GitHub ✅ (`SahilSoftdotin/vision-digital-labs`)
- Your **Supabase** connection details (host/user/password) — same ones in
  `backend/config/application.yml`
- Access to your domain's **DNS** (registrar or DNS provider for visiondigitallab.com)
- Pick a **shared revalidation secret** — any long random string. You'll set the
  **same** value on both Render (`REVALIDATE_SECRET`) and Vercel (`REVALIDATE_SECRET`).

**Deploy order:** backend (Render) **first**, then frontend (Vercel) — so the
frontend's build can fetch live content. (If the backend isn't up yet, the
frontend safely falls back to bundled fixtures and refreshes later.)

---

## 1. Backend → Render

### 1a. Create the service from the Blueprint
1. Sign up at **https://render.com** (log in with GitHub).
2. **New ▸ Blueprint** → connect the `vision-digital-labs` repo.
3. Render reads **`render.yaml`** and proposes the `visiondigitallab-api` web
   service (Docker, region *Virginia*, free plan). Click **Apply**.

### 1b. Set the secret env vars (the ones marked "set in dashboard")
In the service's **Environment** tab, fill these (values from your local
`backend/config/application.yml` / Supabase):

| Variable | Value |
|---|---|
| `SPRING_DATASOURCE_URL` | `jdbc:postgresql://aws-1-us-east-1.pooler.supabase.com:5432/postgres?sslmode=require` |
| `SPRING_DATASOURCE_USERNAME` | `postgres.<your-project-ref>` |
| `SPRING_DATASOURCE_PASSWORD` | *(your Supabase password)* |
| `ADMIN_PASSWORD` | *(choose a strong admin password — replaces `admin12345`)* |
| `REVALIDATE_SECRET` | *(your chosen shared secret — same as Vercel)* |

`JWT_SECRET` is auto-generated by Render. `CORS_ALLOWED_ORIGINS`,
`REVALIDATE_URL`, `JPA_DDL_AUTO`, `SEED_ENABLED` are preset in `render.yaml`.

### 1c. Deploy & verify
- First build takes a few minutes (Docker build + Maven). Watch the logs for
  `Started BackendApplication`.
- Render gives a URL like `https://visiondigitallab-api.onrender.com`. Test:
  - `…/actuator/health` → `{"status":"UP"}`
  - `…/api/v1/services` → JSON list (seeded on first boot)
  - `…/swagger-ui.html` → API docs
- **Free-plan note:** the service **sleeps after ~15 min idle**; the next request
  cold-starts in ~30–60s. Fine here — content pages are cached on Vercel; only
  form submits / revalidation hit the backend.

### 1d. Custom domain
- Render service ▸ **Settings ▸ Custom Domains ▸ Add** → `api.visiondigitallab.com`.
- Render shows a **CNAME target** (e.g. `visiondigitallab-api.onrender.com`).
  Add that DNS record in step 3.

## 2. Frontend → Vercel

1. Sign up at **https://vercel.com** (log in with GitHub).
2. **Add New ▸ Project** → import the `vision-digital-labs` repo.
3. **Important (monorepo):** set **Root Directory = `frontend`**. Vercel then
   auto-detects Next.js (build `next build`, no extra config).
4. **Environment Variables** (Project Settings ▸ Environment Variables):

   | Variable | Value | Notes |
   |---|---|---|
   | `NEXT_PUBLIC_API_URL` | `https://api.visiondigitallab.com` | build-time; the site reads content from here |
   | `REVALIDATE_SECRET` | *(same secret as Render)* | runtime; verifies revalidation calls |

5. **Deploy.** Vercel builds and gives a `…vercel.app` URL to preview.

### 2a. Custom domain
- Project ▸ **Settings ▸ Domains ▸ Add** → `visiondigitallab.com` and
  `www.visiondigitallab.com` (Vercel will set one as primary and redirect the other).
- Vercel shows the exact **A / CNAME** records to add — use those in step 3.

> Tip: until DNS is live you can test against the `…vercel.app` URL. If you do,
> temporarily add that URL to Render's `CORS_ALLOWED_ORIGINS` so the browser calls
> aren't blocked.

## 3. DNS records (at your domain provider)

Add the records each platform shows you. Typical values:

| Host | Type | Points to | For |
|---|---|---|---|
| `@` (apex) | `A` | `76.76.21.21` *(Vercel shows the exact IP)* | Vercel frontend |
| `www` | `CNAME` | `cname.vercel-dns.com` | Vercel frontend |
| `api` | `CNAME` | `visiondigitallab-api.onrender.com` *(Render shows exact)* | Render backend |

DNS can take minutes to a few hours to propagate. Vercel/Render auto-issue HTTPS
certificates once the records resolve.

## 4. Env var summary

**Render (backend)** — secrets in dashboard, rest from `render.yaml`:
```
SPRING_DATASOURCE_URL / USERNAME / PASSWORD   (Supabase)
JWT_SECRET                 (auto-generated)
ADMIN_EMAIL / ADMIN_PASSWORD
CORS_ALLOWED_ORIGINS = https://visiondigitallab.com,https://www.visiondigitallab.com
REVALIDATE_ENABLED = true
REVALIDATE_URL = https://visiondigitallab.com/api/revalidate
REVALIDATE_SECRET          (== Vercel)
SEED_ENABLED = true
```

**Vercel (frontend):**
```
NEXT_PUBLIC_API_URL = https://api.visiondigitallab.com
REVALIDATE_SECRET          (== Render)
```

## 6. Going forward
- **Updates:** `git push` → Render rebuilds the backend, Vercel rebuilds the
  frontend. Automatic.
- **Security before real traffic:** change `ADMIN_PASSWORD` from the default,
  keep `JWT_SECRET` private, rotate the Supabase password if it was shared.
- **Known free-tier limits:**
  - Render free sleeps after idle (cold starts). Upgrade to a paid instance or add
    an uptime pinger to keep it warm.
  - Supabase free pauses after ~1 week idle — a paid plan or periodic ping avoids it.
  - Vercel **Hobby** is free but intended for non-commercial use; a real business
    site should move to **Pro** ($20/mo) eventually.
- **Email lead notifications** are coded but disabled (`NOTIFY_ENABLED=false`).
  To enable later: set `NOTIFY_ENABLED=true` + the `MAIL_*` / `NOTIFY_*` vars on
  Render (Gmail/Workspace App Password). See `backend/.env.example`.
